- Summary
  - The new VERS documentation has been published on the website, transitioning the old RST format to new "introduction" and "specification" documents.
  - Discussions regarding the git PURL type are progressing well, with plans to wait for a draft document detailing background and how-to information before altering the schema.
  - Discussions on SCID namespace focused on issue 516 (https://github.com/package-url/purl-spec/issues/516), concerning namespace qualifiers, where a proposal for a 3-segment namespace was rejected. Consensus was reached to adopt a simpler 2-segment namespace (domain and supplier/publisher) for improved stability and alignment with the SWID standard.
  - The group agreed on an optional registration combined with a `.well-known` convention to manage metadata lookups and improve discoverability for open source projects. The participants were asked to contribute any SCID naming alternatives they might have to issue 835 (https://github.com/package-url/purl-spec/issues/835).
- **Approval of Previous Meeting Minutes and Code of Conduct**: Philippe Ombredanne opened the meeting by reminding attendees about the code of conduct, which is available on the TG2 web repository on GitHub, and encouraged everyone to be nice to each other. The meeting minutes from the previous meeting on February 27th were approved.
- **Agenda Setting and Roll Call**: The initial agenda items included the continuing discussions on the 'scid' and 'git' types, proposed by Michael Herzog. Steve Springett added a brief context item on populating the VERS specification with Ecma markup and a discussion regarding Tom Alrich's PURL expansion work, for which Steve volunteered to be a co-lead.
- **VERS Documentation Updates**: Michael Herzog reported that the new version of the VERS documentation has been published on the website, with the VERS section now broken into multiple documents. The new documents, "introduction" and "specification," are intended to serve as draft content for the standard, in parallel with what was done for PURL with ECMA 427 (https://ecma-international.org/publications-and-standards/standards/ecma-427/). The old RST documentation has been updated to redirect users to the website, effectively retiring the old format.
- **Progress on Git PURL Type Discussion**: There has been active discussion regarding the git PURL type, which Michael Herzog described as "almost ready". A separate point of discussion that emerged was where to place background or "how-to" information about a specific PURL type, which should not be part of the schema. Michael Herzog suggested waiting for John Horan to prepare a draft of this additional documentation, which could be accommodated without altering the definition schema.
- **Focus on SCID Namespace Discussion**: The main driving issue for the scid discussion is issue 516 (https://github.com/package-url/purl-spec/issues/516), which Michael Herzog noted is focused primarily on the namespace aspect. The prior consensus was to focus first on namespace, as there are no structural decisions required beyond coming up with a set of recommended or registered qualifiers. The current item on the table is determining the required namespace for this new PURL type.
- **Proposed Three-Segment Namespace for SCID**: The original proposal suggested a "domain/vendor" namespace, which Michael Herzog suggested changing to "supplier or publisher," a change that appeared to be agreed upon. The major consideration is whether to include a third segment, such as a supplier-specific namespace or product line, to create a three-segment namespace (domain, supplier, and supplier sub-namespace).
- **Rejection of a Three-Segment Namespace**: Steve Springett was not in favor of adding business units or product lines to the namespace, arguing that existing mechanisms like subdomains and sub-paths can handle this context. Philippe Ombredanne concurred, as adding a third segment complicates the default format. Piotr P. Karwasz agreed, noting that making the namespace dependent on organizational structure, like a business unit, would make the PURL less stable if products change ownership.
- **Consensus on a Two-Segment Namespace**: The consensus was against the proposal for a three-segment namespace, which was confirmed as a "no" by Michael Herzog. The preference is to keep the namespace definition simpler, using only two segments. This approach prevents unnecessary parsing for empty segments and allows for future extensions based on experience.
- **SCID Namespace Design and CPE Integration**: The recommendation for a two-segment namespace (domain and supplier/publisher) was directly influenced by the SWID standard, which already has consensus from other communities. The goal is to use this framing and then determine the PURL-specific details and gaps, such as integrating necessary information for CPE (Common Platform Enumeration) compatibility. The desire to fully replace CPE functionality requires the ability to capture the publisher/supplier information.
- **Concerns about Encoding Requirements**: Philippe Ombredanne expressed concern about encoding requirements for slashes in the namespace, wishing for a future version of the PURL specification to potentially remove encoding requirements for all slashes. A major topic of discussion was whether the publisher/supplier segment should be mandatory.
- **Mandatory vs. Optional Publisher/Supplier Segment**: Steve Springett argued for making the publisher mandatory to prevent potential name confusion for projects hosted on platforms like GitHub that don't have their own domain. Michael Herzog noted that the domain and supplier parts are crucial for the commercial/proprietary side, while Philippe Ombredanne argued against making the supplier mandatory for projects like Linux to avoid the burden of inventing a supplier if one is not naturally recognized.
- **Registration and Conformance Testing**: Michael Herzog stated that, from a specification perspective, registration should not be a requirement for base conformance. Conformance testing should only require syntactic accuracy, while advanced tests or tools could check for registration and provide warnings or messages. Steve Springett supported making the barrier to entry for this PURL type simple to avoid hurting adoption, referencing the complicated distribution requirements of CSAF.
- **Registry vs. Automated DNS/Well-Known Lookup**: Steve Springett proposed that instead of maintaining a central registry, the system could use DNS checks and a "well-known" mechanism to automatically look up contact information and metadata. This allows organizations to maintain control over their information, including contact details, which helps with security responsiveness. Michael Herzog acknowledged that the DNS records are not always reliable for providing contact information due to privacy measures.
- **Proposal for Optional Registration and Well-Known Convention**: Philippe Ombredanne suggested a combination approach: having an optional registration for discoverability and a well-known convention that serves the same information but is distributed. The registration could act as a bootstrap mechanism or an aggregation point, while the well-known location, possibly an RFC like 9472 (https://datatracker.ietf.org/doc/html/rfc9472), would allow querying for the same metadata. Michael Herzog and Steve Springett agreed that the registration is likely more beneficial for open source projects, while commercial entities will want full control over their data.
- **Use of TC54 Standards for Metadata**: The conversation touched upon using other TC54 standards for deeper data retrieval, such as CSAF and NT (Notification Type). Steve Springett suggested that whatever schema is decided upon for the PURL metadata, it could point to the TEA-server or the CLE so users can dive deeper if needed. Michael Herzog cautioned against PURL covering too much territory, advocating for using the right tool for the job.
- **Next Steps for SCID Namespace**: Philippe Ombredanne committed to developing a suggestion for the namespace structure that accommodates both optional registration and a .well-known/ convention. Michael Herzog indicated they would create a separate issue to focus specifically on the registry side of things to break out the discussion. The goal is to support the minimal use case of namespace, name, and version without additional requirements like dealing with events.
- **Reviewing Display Options for Package Information**: Piotr P. Karwasz and Philippe Ombredanne discussed the display options for package details, which Philippe Ombredanne demonstrated using examples of kernel and project information. Philippe noted that users could opt to include names, descriptions, version lists for each package (e.g., "logging solar"), or choose to provide no names at all. They also discussed displaying project information depending on what the user elects to do.
- **Reviewing Logging Project Data and Update Process**: Philippe Ombredanne drilled down on the logging information, which Piotr P. Karwasz noted is based on a DOAP (Description Of A Project) file that is not updated. Piotr P. Karwasz explained that Apache trusted releases should be responsible for updating those files. The logging section includes three projects, and Philippe Ombredanne pointed out that the log4j entry is present but not explicitly named Apache.
- **Discussing Naming for Software Component ID**: The participants discussed improving the naming of the concept currently referred to as "scid," or Software Component ID. Michael Herzog noted that searches for "skid" do not predominantly return negative connotations, although "skid mark" and "Skid Row" appear. Philippe Ombredanne suggested alternatives like "SPeeD" (Software Package Extended Enumeration) or "SPID" (Software Package ID) (see issue 835 – https://github.com/package-url/purl-spec/issues/835), but they agreed to dive into how to structure and decide on a better name.
- **Assignment for Name Brainstorming on Ticket 835**: Michael Herzog confirmed that ticket 835 is dedicated to brainstorming names and encouraged Philippe Ombredanne and Piotr P. Karwasz to contribute ideas. Michael Herzog added that the name does not necessarily have to be limited to four characters. The participants agreed to gather ideas on the issue, emphasizing that all ideas are welcome in a brainstorm, provided they are not offensive.
- **Developing a Practical Example for Apache Identifiers**: Philippe Ombredanne suggested that they need to develop a practical example of the system specifically for Apache. Piotr P. Karwasz confirmed they intend to use the set of identifiers built by Apache trusted releases for projects like logging and log4j.
- The meeting was adjourned.
