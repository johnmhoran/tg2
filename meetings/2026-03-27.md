## Notes

- The meeting is being recorded.  Our code of conduct applies to this meeting.  https://github.com/Ecma-TC54/tg2/blob/main/CODE_OF_CONDUCT.md.

- Summary
  - A new process for major decisions was adopted, utilizing a document template and requiring changes to be submitted via pull request. All participants agreed to emphasize the use of GitHub Discussions for proposal commentary due to their threaded features, which makes topics easier to follow.
  - The team decided to proceed with a single PURL type for 'sid' (Software IDentification), managing the distinction between open-source and commercial software through existing or expanded PURL qualifiers. Documentation for CPE equivalency was decoupled from the PURL type definition and will be addressed in a separate guidance document.
  - The requirement for namespace registration was clarified, stating that a registry-based namespace not registered in the PURL registry is invalid. Focus shifted to defining comprehensive documentation on the namespace for the new 'sid' PURL type, guiding human users.
- **Meeting Commencement and Minutes Review**: The meeting began with a reminder to adhere to the Ecma-TC54 code of conduct. Philippe Ombredanne summarized the code as "Just be nice to each other". The minutes from the previous call were presented, and no objections were raised.
- **Agenda and Roll Call**: A quick roll call was conducted to review the agenda, confirming the presence of Michael Herzog, Steve Springett, and John Horan. Philippe Ombredanne noted that the topics Michael Herzog brought up were also of interest to them.
- **Proposed New Decision/Major Topic Format**: Michael Herzog proposed a new format for handling major, long-running decisions, which traditionally used issues in the repository. The new approach involves using a document template, distinct from existing documents, for proposals regarding decisions.
- **Process for Handling New Decision Proposals**: The new process dictates that changes to the proposal document should be submitted via a pull request, and commentary on the proposal should be handled in a dedicated discussion item. Michael Herzog explained that discussions are preferred over issues because they allow for threaded comments, aiding in the consolidation of commentary and making it easier to track specific topics. The participants agreed to emphasize the use of GitHub Discussions for general ideas or topics that are not ready for a formal decision.
- **Process Changes and Closure Goal**: Michael Herzog reinforced that these process changes are essential for achieving closure on major topics, noting that issues are not the ideal place for design discussions. They mentioned that they did not choose the term "designs" but were guided towards the Architectural Decision Record (ADR) approach by Steve Springett, even though the standard is not a traditional software implementation.
- **Decision Document Structure and Metadata**: The decision documents are intended to be simple, receiving a number, and containing a status and a date at the top. The proposed structure avoided including extensive metadata such as details on who was consulted and when, although that information could be added if requested by participants.
- **Repository Documentation Structure**: Michael Herzog introduced the use of a README file in a directory to explain the contents for new users, which helps avoid cluttering the main project README file. This structural approach was used for the `docs/` and `decisions/` folders.
- **Adoption of New Process and Benefits**: Philippe Ombredanne expressed strong support for the proposed changes, stating that they make a lot of sense and will help declutter issues and pull request commands to make discussions more effective and efficient.
- **Contributing File and AI Content Policy**: Michael Herzog created a new `contributing` file that includes a mild anti-AI policy, requiring that all written content in a pull request must be human-authored and mandating the disclosure of any AI usage, including the tools and how they were used. This policy aims to address the growing influx of AI-generated content.
- **Debate on Single vs. Multiple Package-URL (PURL) Types**: The discussion moved to whether a single new PURL type should cover both open-source and commercial/proprietary software, or if the effort should be split into two types. Michael Herzog expressed concern that trying to cover too much with one type might slow down consensus and completion.
- **Consensus on Single PURL Type for SID**: Steve Springett argued against multiple PURL types, suggesting it would lead to confusion and duplication. They proposed that distinguishing features between open-source and commercial aspects should be managed through existing or expanded PURL qualifiers, which are applicable to many ecosystems. The consensus reached was to proceed with a single PURL type, 'sid' (Software IDentification), and manage distinctions through qualifiers.
- **Managing Qualifiers and CPE Equivalency Documentation**: The team decided to refine the documentation surrounding PURL qualifiers. The idea of achieving CPE equivalency was decoupled from the PURL type definition itself and will be addressed in a separate guidance document or FAQ page on the website.
- **Focus on Namespace for SID Type**: With the qualifier and CPE issues separated, the discussion's focus returned to defining the namespace for the new 'sid' PURL type. The plan is to develop comprehensive documentation on the namespace to guide human users, in addition to the concise definition needed for the specification.
- **Clarity on Namespace Registration Requirement**: The discussion addressed the requirement for namespace registration, particularly for non-domain-based namespaces. The current proposal states that a registry-based namespace not registered in the PURL registry is considered invalid, providing clarity for tool creators.
- **Domain-Based Namespace and Validation Challenges**: The participants discussed the challenges of enforcing validation when a company uses a domain-based namespace (e.g., `servicenow.com` PURLs) but chooses not to publish .well-known files or register the namespace. It was acknowledged that while this is not ideal, they cannot prohibit internal or non-public usage, and the community may need to gently reach out to organizations to encourage publishing .well-known files.
- **Registries and Registration Philosophy**: The conversation branched into the broader philosophy of PURL type registration, suggesting a more coherent strategy is needed for who has authority over the registries. Currently, the process relies on checking if the proposer of a new PURL type has a connection to the relevant community.
- **Linux Kernel Use Case for Namespace**: In response to Steve Springett's question regarding the Linux kernel, Philippe Ombredanne confirmed that the Linux Kernel representatives prefer a domain-based namespace, specifically `kernel.org`, with the intention of hosting their own well-known records for autonomy. They expect the Linux team to focus first on `kernel.org/Linux/`.
- **Testing and Baseline Requirements for PURLs**: Michael Herzog reiterated the need for clear criteria for PURL definitions to ensure tool support. The baseline requirement for non-domain-based namespaces would be that they must be registered.
- **Path Forward for SID PURL Type**: The team agreed that the 'sid' PURL type is generally well-defined, but the process for resolving the domain versus registry location for the namespace still needs refinement. Michael Herzog proposed keeping the namespace rules and registries separate from the PURL type definition.
- **Handling of Existing Namespace-Related PRs**: Michael Herzog noted that pending pull requests for other PURL types, such as RPM and Deb, which involve simpler namespace registration concerns, should be addressed after the SID framework is merged. The approach will be to apply the new, refined mechanism developed for SID to these existing types.
- **Decision Document Maintenance**: The team confirmed that the decision document for SID will be updated via pull requests, serving as a record of the agreed-upon direction. This will then be followed by more narrowly focused issues and pull requests to implement the changes.
- **Value of GitHub Discussions**: Steve Springett liked the idea of using GitHub Discussions, though they noted that most tools, including one they built, do not currently support them. Michael Herzog maintained that discussions are the right tool for the job, and the project should adopt them.
- **Meeting Conclusion and Next Steps**: The meeting concluded with a summary of the path forward: retaining one SID PURL type, separating concerns while working on them in parallel, and focusing on the namespace registration issues first. The team agreed they had a path forward.
- The meeting was adjourned.
